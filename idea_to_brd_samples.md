# Idea-to-BRD Sample Inputs

Use these samples in the `Idea to BRD` workspace module to test Groq-backed BRD generation. Each sample matches the current UI fields.

## 1. AI Campus Helpdesk

**Idea Title:** AI Campus Helpdesk

**Raw Idea:** Build a web app for college students to ask questions about admissions, exam schedules, fees, hostel rules, placement deadlines, and department contacts. The system should answer common queries instantly, allow students to raise unresolved requests, and let administrators update official answers from one dashboard.

**Target Users:** Students, admission staff, department coordinators, placement cell, hostel office, and college administrators.

**Business Goals:** Reduce repetitive support queries, improve response time for students, centralize official information, and track unresolved requests.

**Constraints:** Must work well on mobile, protect private student data, support authenticated admin updates, and maintain an audit trail for changed answers.

## 2. Restaurant Inventory Control

**Idea Title:** Smart Restaurant Inventory Control

**Raw Idea:** Create a restaurant inventory system that tracks ingredients, daily stock usage, purchase entries, supplier details, and low-stock alerts. Managers should be able to see which ingredients are running out and estimate how long current stock will last based on recent sales patterns.

**Target Users:** Restaurant owners, kitchen managers, procurement staff, and accountants.

**Business Goals:** Reduce ingredient wastage, avoid stockouts, improve purchase planning, and give owners clearer visibility into inventory cost.

**Constraints:** Should support simple data entry from mobile devices, work for small restaurants with limited technical staff, and avoid complex ERP-style workflows.

## 3. Clinic Appointment Booking

**Idea Title:** Clinic Appointment Booking Portal

**Raw Idea:** Build a clinic appointment portal where patients can search available doctors, book appointments, reschedule visits, receive reminders, and view appointment status. Clinic staff should be able to manage doctor availability, confirm walk-ins, and update patient queues.

**Target Users:** Patients, reception staff, doctors, and clinic administrators.

**Business Goals:** Reduce phone-based booking load, decrease missed appointments, improve queue visibility, and make doctor availability easier to manage.

**Constraints:** Must be easy for non-technical patients, avoid exposing medical details unnecessarily, and support appointment reminders through future notification integrations.

## 4. Local Delivery Tracker

**Idea Title:** Local Delivery Tracker

**Raw Idea:** Create a lightweight delivery tracking system for small local businesses. Store owners can create delivery orders, assign them to riders, update order status, and share a tracking link with customers. Riders should see assigned deliveries and mark pickups and drop-offs from their phones.

**Target Users:** Local shop owners, delivery riders, dispatch coordinators, and customers.

**Business Goals:** Improve delivery visibility, reduce customer calls about order status, help owners manage riders, and keep delivery operations simple.

**Constraints:** Must support low-end mobile phones, should not require a dedicated customer app, and should handle intermittent rider connectivity gracefully.

## 5. Event Registration Portal

**Idea Title:** Event Registration Portal

**Raw Idea:** Build a portal for colleges and clubs to create events, collect participant registrations, approve attendees, generate QR-based entry passes, and track attendance on event day. Organizers should be able to export participant lists and see registration counts by event.

**Target Users:** Event organizers, student clubs, college departments, participants, and entry desk volunteers.

**Business Goals:** Simplify event registration, reduce manual spreadsheet work, improve attendance tracking, and give organizers clean participant data.

**Constraints:** Must support QR scanning from mobile devices, avoid duplicate registrations, work for multiple events, and export data in a format organizers can share.
